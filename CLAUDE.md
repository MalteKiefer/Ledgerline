# Ledgerline — Projekt-, Anforderungs-, Security- & Session-Kontext

Self-hosted **zero-knowledge personal cloud** (Laravel). Server hält NUR Ciphertext:
alles wird im Browser ver-/entschlüsselt. Selbst der Server-Betreiber kann Daten
nicht lesen. Single-tenant Server, aber code-seitig **voll Multi-User-isoliert**.

Module: **Dashboard, Galerie, Dateien, Notizen, Todos, Lesezeichen, Passwörter (inkl. `passkey`-Typ
+ eingebettete Passkeys in `login`-Items), Kontakte, Rechnungen, Health, Explore (Karte + GPS-Tracks), Backup, Paperless**.
Version **v1.505.47** (live https://home.kiefer-networks.de, `/up`=200).
Zusätzlich: **Browser-Extension** (Chromium, MV3) für ZK-Passwort-Autofill + Bookmarks-CRUD
+ **WebAuthn-Authenticator** (Passkeys).

`README.md` ist die maßgebliche, gepflegte Feature-Beschreibung. Diese Datei =
Arbeitskontext + stehende Anweisungen + **Security-Entscheidungsprotokoll**.
Detail-Historie in `~/.claude/projects/-Users-malte-kiefer-Entwicklung-ledgerline/memory/`.

## Stack
- Laravel 13 / PHP 8.5 (`declare(strict_types=1)`), PostgreSQL 17 **pgvector** (prod) / sqlite (tests), Valkey 8 (predis, kein phpredis).
- Frontend: **Alpine.js modularisiert (Vite-gebündelt)** — `resources/js/app.js` (~753 Z., Bootstrap + Store-/Komponenten-Registrierungen) + `resources/js/shared/*` (Utilities) + `resources/js/components/*` (ein Modul pro Alpine.data-Komponente) + Blade + Tailwind 4 + Vite 8. Kein externes CDN. **Konvention „Alpine in EINER Datei" ist aufgehoben** (Refactor abgeschlossen, s. REFACTORING).
- **Client-Crypto** `resources/js/vault.js`: Passphrase --Argon2id(ops=4/mem=256MiB)--> KEK --unwrap--> Vault Key (VK). Sealing via libsodium-wrappers-sumo (XChaCha20-Poly1305 secretbox). Wrap der VK auf Trusted-Device via AES-256-GCM (WebCrypto, non-extractable, in IndexedDB). Entschlüsselung schwerer Blobs im Worker-Pool `resources/js/decrypt.worker.js`.
- Auth: Pocket-ID OIDC (Socialite `pocketid`, **PKCE + state**), Match auf stabiler `sub`. Keine App-Passwörter. `groups`-Claim → Admin-Gate (fail-closed bei Multi-User ohne Gruppe).
- Sanctum 4 Bearer-Tokens für Mobile/CLI/Extension-API (`/api/v1`). sabre/dav v4 NUR für Files-over-WebDAV + WebDAV-Backup-Ziel (kein CalDAV/CardDAV). Bilder: intervention/image v4 + imagick. ML: immich-machine-learning Sidecar (profile-gated, optional).
- **Dependency-Policy:** immer neueste stabile Version, per exakter Version + Digest gepinnt (Docker-Images), keine Alt-Majors aus Bequemlichkeit. CI (Dependabot + Trivy) hält das nach. Stand: PHP 8.5, Node 22 LTS, Laravel 13.21, PHPUnit 13.2, guzzle 7.15.1, pdfjs-dist 6.x, dompurify 3.4.12, vite 8.1.5, marked 18.0.7, uplot (exakt gepinnt), fflate 0.8.3 (KMZ-unzip; Explore-Karte nutzt Leaflet+OSM wie der Location-Picker), concurrently 10.0.3 (build-dev, +`shell-quote`-override), rector/rector + driftingly/rector-laravel + cyclonedx/cyclonedx-php-composer (dev-only). **Transitive Deps zählen mit** — ein rot gewordener `composer audit`/`npm audit` wird als eigener Patch-Release behoben, nicht ausgesessen (s. v1.504.49).

## Alpine-Gotchas
- `<template x-if>` braucht genau 1 Root-Element. Objekt-`:style` statt String-`:style`. `x-for` ebenfalls Single-Root.
- Kein nackter JS-Kommentar als Alpine-Ausdruck. Bei Teardown transient `null` → Templates mit `?.`/Null-Guards absichern (z. B. `draft?.fields?.urls`).
- Closure-Argumente von `Alpine.data(...)` sind in Blade-Ausdrücken NICHT sichtbar → `{{ __('...') }}` statisch oder `@js(__('...'))`. `@js()` NUR in Blade, nie in app.js.
- `<x-icon ::name>` reaktiv geht NICHT (server-gerendert) → per-Wert-Icons via x-show-Toggle (`passwords/_icon.blade.php`).
- `$store.vault.unlocked` ist eine **Property**, keine Methode (Files-Init-Crash v1.504.38).
- **Store-abgeleitete Getter** (Komponenten, die aus `LLModuleStore.<m>.data`/`LLFilesStore.data`/`LLGalleryStore.data` lesen) müssen `void this._mut` referenzieren — diese `data`-Objekte sind **kein** Alpine-reaktives Objekt, sonst aktualisiert sich das UI nicht (Dashboard v1.504.46).
- **Alpine-Komponenten (Stand v1.505.8, 19):** `dashboard`, `vaultGallery`, `vaultFiles`, `notes`, `todos`, `bookmarks`, `contacts`, `invoices`, `passwords`, `health`, `explore`, `backupRuns`, `devicePairing`, `paperlessSettings`, `notificationBell`, `cropModal`, `toastHub`, `publicShare`, `fileShare`.

## ZERO-KNOWLEDGE-ARCHITEKTUR (Kern)
- **Key-Hierarchie** (`vault.js`): Passphrase → Argon2id(ops=4/mem=256MiB) → KEK; KEK unwrapped per-user **Vault Key (VK)**. VK verlässt Browser nie. Recovery-Key als zweiter Wrap (32 Byte Zufall → generichash). Trusted-Device: VK mit non-extractable AES-256-GCM-Key in IndexedDB gewrapped (`VAULT_REMEMBER_DAYS`, frische IV je Wrap, Owner-Binding); Public-Computer: session-only + Idle-Lock (`VAULT_PUBLIC_IDLE_MINUTES`).
- **Per-Modul Sealed Stores (Store v3, v1.505.0):** JEDES Modul hat sein EIGENES opakes sealed Store — kein Monolith mehr. **Notes/Todos/Bookmarks/Contacts/Invoices/Passwörter/Health/Sharing** liegen je als eine Zeile in **`module_stores`** (composite PK `(user_id, module)`, sealed ciphertext + version, `ModuleStore`/`ModuleStoreController`, MODULES-Allowlist → 404 bei unbekanntem Key, ETag/304 + optimistic 409). **Files** hat ein EIGENES **sharded** sealed Store (`files_store`, `FilesStore`/`FilesStoreController` via `SealedManifestStore`, Store v3 §4.2 — root pointer table + content-addressed id-bucketed shard blobs + fileFolders-Collection-Blob, spiegelt `GalleryStore`). Galerie unverändert eigenes sharded Manifest (`GalleryStore`). **Ein Write in einem Modul re-sealt NIE die anderen** (Kern-Motivation des Splits). Der Monolith `VaultStore`/`vault_store`/`StoreController`/`/store` ist ENTFERNT (clean-slate, kein Back-Compat, Datenverlust der Alt-Monolith-Blobs akzeptiert — Client baut jedes Modul-Store beim ersten Save neu). Client: `window.LLModuleStore.<module>` (Registry aus `shared/module-store.js`, per-Modul `load/touch/flush/reset/newId`), `window.LLFilesStore` (sharded, in app.js), `window.LLGalleryStore` (sharded). Der `sharing`-Store hält `knownFingerprints` (TOFU), von Passwörtern + Files gemeinsam genutzt.
  - **API-Mapping (Mobile, ZK):** die RECORDS von **Notes, Todos, Bookmarks, Contacts, Invoices, Passwörter, Health, Sharing** werden ausschließlich über **`GET/PUT /api/v1/store/{module}`** gepflegt (Modul-Store lesen → client-seitig mutieren → zurückschreiben, optimistic version); **Files** über **`GET/PUT /api/v1/files/store`** (sealed sharded root). **Kein per-Record-Endpunkt** — ein solcher bräche ZK. **Notes + Todos sind reine Store-Module** (keine Blobs, keine eigene Route). Nur BINÄR-Blobs haben eigene Routen: **Contact-Avatare** `GET /api/v1/contacts/usage`, `POST /api/v1/contacts/upload`, `GET /api/v1/contacts/raw/{blob}`, `DELETE /api/v1/contacts/blob/{blob}`, `POST /api/v1/contacts/blobs/reconcile` (selber `ContactBlobController`, Sanctum `abilities:device`) — die Kontakt-Datensätze selbst liegen im `contacts`-Modul-Store, die Avatar-Routen halten nur die optionalen Bild-Bytes. Files/Gallery-Blobs analog. In `openapi.yaml` unter Tags **Store** (`/store/{module}`) + **Files** (`/files/store`) + **Contacts** dokumentiert.
  - **Vollständige `/api/v1`-Fläche (v1.504.44, „Apps nur via API" — `openapi.yaml` ist die maßgebliche, 1:1 mit `route:list --path=api/v1` verifizierte Referenz, 84 Ops):** **Auth/Pairing** (`auth/pair`+`collect`, `me`, `avatar`, `device/heartbeat` {state,detail}→{wipe}, `auth/session` DELETE). **Store** (`GET/PUT /store`). **Files** (usage/reconcile/upload+chunked/raw/blob-delete + **shares** store/update/destroy). **Gallery** (store, usage, reconcile, upload+chunked, raw, blob-delete, **process/analyze/embed-text** ML, **reverse**+**geocode** Geocoding, **shares** store/update/destroy). **Contacts** (Avatar-Blobs + `contacts/notify`). **Passwords-Enrichment** (icon/breach/tfa-directory `{entries}`). **Vaults** (keys, vaults CRUD, store, resolve-recipient, members+accept, rotate {members[].user_id + remove_member_id}, DELETE→204, index liefert `owner`) + **`vaults/{v}/blobs/*`** (shared-folder blobs). **Geräte** (`GET /devices`, `DELETE /devices/{token}`, `POST /devices/{token}/wipe` Remote-Wipe). **Notifications** (index ETag/304, {n}/read, read-all). **Account/DSGVO** (`account/export` Stream, `account` DELETE {confirmation}, `account/sessions/{id}` DELETE). **Präferenzen** (`profile/avatar/refresh`, `locale`, `theme` — die Web-Redirect-Controller antworten via `expectsJson()` mit JSON). **Bewusst NICHT auf `/api/v1`** (admin/single-tenant, nur Web+Console): **Backup, Paperless-Config, Workspace-Settings** (company/security/system/notifications-config). Jede API-Änderung ⇒ `openapi.yaml` im selben Commit.
- **Blob-Ledger + Modelle:** `GalleryBlob`, `FileBlob`, `ContactBlob`, `SharedFolderBlob` (+ Sealed-Stores `ModuleStore`, `FilesStore`, `GalleryStore`, `SharedVaultStore`). Bytes sind client-Ciphertext, Server streamt roh. Blobs zu Padmé-Buckets gepaddt, Manifest Padmé mit 4 KiB-Floor. **Padmé gilt auf ALLEN Write-Pfaden inkl. Extension** (`extension/src/crypto.js sealManifest` spiegelt `vault.js`).

- **Sharing (öffentliche Links, ZK):** `/s/{token}` für Galerie-Album, Datei UND Ordner — Share-Key im URL-Fragment (nie an Server), optionales Passwort = rate-limitiertes Gate, optionaler Download, Expiry. `PublicShare`-Modell (`kind`), Trait `Concerns\ManagesPublicShares`, `PublicShareController` (public), `ShareCrypto` in vault.js. **`ShareCrypto` ist ein bewusst SEPARATES, self-contained SYMMETRISCHES Format** (per-file key gewrappt unter dem Fragment-Share-Key als schlichtes `{c,n}`-secretbox-JSON — XChaCha20-Poly1305) — NICHT der Store-v3-Suite-/canonical-JSON-Envelope (`{suite,epk,kem_ct,c,n}`) und KEIN PQ-hybrid-KEM (public Links haben keinen Empfänger-Public-Key; der Fragment-Key ist der einzige Schlüssel). Public-Share-ZK-Seiten brauchen das gebündelte JS (globale CSP `script-src 'self'` reicht).
- **Cross-User-Ordner-Sharing (ZK, v1.504.35):** echtes Teilen von ORDNERN mit registrierten Nutzern — **generalisiert auf dem Passwort-Tresor-Sharing**. Ein geteilter Ordner = ein `SharedVault` mit **`kind='folder'`** (Spalte `shared_vaults.kind`, default `password`, server-assigned, nie fillable); sein sealed Manifest (`SharedVaultStore`) trägt `{name, folders[], files[]}`; Datei-Blobs unter **VK_folder** verschlüsselt (per-file key gewrappt via `Vault.encryptContentWith(bytes,meta,VK_folder)`), im eigenen Ledger `shared_folder_blobs` (`blob` PK, `vault_id` FK cascade, `owner_id`=Ordner-Owner für Quota, `size`), Disk-Prefix `shared-folders/{blob}`. **Member-scoped Blob-Zugriff:** `SharedFolderBlobController` (extends `BlobStoreController`) — `view`=lesen/usage/reconcile (Rolle ≥ viewer), `update`=upload(single+chunked)/delete/reconcile (Rolle ≥ editor), gated via `SharedVaultPolicy` (fail-closed, denyAsNotFound). Quota gegen **Owner-Total** (`FileBlob` + `shared_folder_blobs` des Owners vs `files.quota_mb`, 413). Routen `/vaults/{vault}/blobs/{upload,upload/init|part|complete|abort,raw/{blob},{blob} DELETE,usage,reconcile}` auf WEB (`vaults.blobs.*`) + `/api/v1` (`api.vaults.blobs.*`), gleiche Throttles wie persönliche File-Blobs. ## THREAT MODEL (STRIDE + LINDDUN)
Formales Bedrohungsmodell für diese ZK-App (§14). Ergänzt die Adversary-ASSUMPTIONS um eine strukturierte Kontroll-↔Residual-Matrix. Konkrete Mechanismen dieser App, kein Generik. Kern-Annahme: **Server (App/DB/Blob/Backup) ist honest-but-curious → sieht NUR Ciphertext + Version + gepaddete Größen.** Was der Server nie sieht: VK/VK_folder/VK_vault, Passphrase, Klartext-Inhalt, wrapped Key-Material im Klartext, Share-Key/Passphrase (URL-Fragment).

### Trust Boundaries
- **Web-Client (Browser):** Krypto-Grenze — Passphrase→Argon2id→KEK→VK, alle Seal/Unseal, DOMPurify-Markdown, Trusted-Device-VK in IndexedDB (non-extractable AES-GCM). Innerhalb dieser Grenze existiert Klartext.

- **`/api/v1` + Web-Server (ZK-Relay):** Sanctum (`abilities:device`) / Session, Owner-Scope, Policies. Relayt Bytes, liest sie nie.
- **DB + Blob-Store (opaker Ciphertext at rest):** PostgreSQL + Blob-Disk/S3 — sealed Manifeste (`module_stores`/`files_store`/`gallery_store`/`shared_vault_stores`), Ledger-Blobs, wrapped Keys. Padmé-gepaddet.

- **ML-Sidecar / Geocode / externe-Fetch-Grenze:** opt-in, transient, SSRF-guarded über `OutboundUrl` (Favicon/BIMI, HIBP, 2fa.directory, Photon, Paperless, ntfy/SMTP). Der EINZIGE Klartext-verlassende Pfad ist das minimierte `/process`-Fenster (Foto-Bytes, opt-in, Temp-unlink).
